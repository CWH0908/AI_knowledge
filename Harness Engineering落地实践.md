# Harness Engineering 落地实践指南

> 核心目标：让 AI 在工程里持续、稳定、规范、顺畅地产出正确结果

**什么是 Harness Engineering**：不是某一个工具，也不是某一条提示词技巧，而是一整套让 AI 在工程里稳定产出正确结果的工程系统。

三个关键词：
- **稳定**：不是这次运气好做对了，而是换需求、换维护人仍能稳定工作
- **产出**：不只是写代码，还包括需求、方案、验证、交付等完整过程产物
- **正确结果**：不是"做完了就算"，而是有办法判断做得对不对

---

## 一、核心概念速览

| 概念 | 解决的问题 | 工程角色 |
|------|-----------|---------|
| Rule | 什么事绝对不能乱来 | 基础规矩、红线、底线 |
| Skill | 这件事具体应该怎么做 | 固定动作的标准操作手册 |
| Sub Agent | 复杂任务由谁分工 | 处理不同阶段的专业角色 |
| Workflow | 这些角色按什么顺序接力 | 前进、暂停、打回、重跑规则 |
| Scripts | 谁来判断到底做没做好 | 统一门禁和事后验证 |
| MCP | AI 怎么安全接上外部系统 | 外接能力与工具链 |

### 1.1 Rule（规则）
- **本质**：给 AI 写的工程规矩，告诉它什么能做、什么不能做
- **作用**：减少低级错误，而非让 AI 变聪明
- **局限**：只是软约束，AI 可能忽略或绕过

### 1.2 Skill（技能）
- **本质**：固定步骤的标准操作手册
- **价值**：让 AI 不要临场发挥，严格按固定步骤执行
- **典型场景**：编译、测试、事后验证

### 1.3 Sub Agent（子代理）
- **本质**：多个分工明确的 AI 角色
- **价值**：避免单 Agent 既做需求、又做开发、又自己审自己
- **核心思路**：每个 Agent 只负责自己那一段

### 1.4 Workflow（工作流）
- **本质**：接力赛规则，不只是流程图
- **关键要素**：
  - 当前任务处于哪个阶段
  - 每个阶段的产出是什么
  - 谁接下一棒、接手前必须看到什么
  - 哪些问题必须打回、回到谁那里

### 1.5 Scripts（脚本）
- **本质**：整个 Harness 里最"硬"的东西
- **价值**：你说做完了没用，得跑过脚本才算
- **意义**：把"是否完成"从主观汇报变成客观检查

### 1.6 MCP（模型上下文协议）
- **本质**：把仓库之外的能力接进 AI 工作链路
- **用途**：对接 CI、签名、制品、发布等外部工程系统
- **定位**：Harness 的外接能力接口，而非主体

### 1.7 概念间的层次关系

```
Rule 负责告诉 AI 什么是底线
    ↓
Skill 负责把高频动作标准化
    ↓
Scripts 负责判断结果到底对不对
    ↓
Sub Agent 负责把复杂任务拆成多个专业角色
    ↓
Workflow 负责让这些角色能按顺序接力
    ↓
MCP 负责把这套系统继续往外部工程系统延伸
```

> 这几个东西不是互相替代的，而是逐层叠加的。

## 二、搭建顺序与步骤

### 2.1 第一步：磨透设计规格文档（SPEC）

> 先把目标、边界、验收标准说清楚，没有这一步，后面所有约束都会飘。

**SPEC 必须包含**：
- 版本要解决什么问题
- 核心目标 vs 顺手优化
- 影响哪些模块
- 兼容要求
- 完成标准

**要求**：没有不确定词语（如"建议"、"可以"、"推荐"、"可选"）

### 2.2 第二步：补关键 Rule

> 只盯最容易反复出错、最容易偷懒的底线

**典型 Rule 示例**：
```
改完代码后必须：
1. 编译
2. 测试
3. 事后验证
三步不全过，开发不算完成
```

**原则**：不要第一天就写几十条，先从最痛点开始

### 2.3 第三步：把高频动作做成 Skill

> 每次都要做、不希望 AI 临场发挥的动作

**典型 Skill**：
- 编译 Skill：找对 MSBuild、固定配置、还原、输出日志、错误模式
- 测试 Skill：标准测试流程
- 事后验证 Skill：统一验证入口

**好处**：
- Rule 变轻（只保留红线）
- AI 执行稳定性变高
- 维护成本降低

### 2.4 第四步：拆分多 Agent（当单 Agent 开始失稳时）

**七个核心 Agent**：

| Agent | 职责 |
|-------|------|
| 项目经理（PM） | 路由、交接、回退、进度管理（只做流程管理，不给专业意见） |
| 需求分析 | 把模糊诉求变成清晰需求 |
| 方案设计 | 把需求变成技术方案 |
| 闸门总控 | 开发前最后的可行性和风险把关 |
| 开发实现 | 落地代码和实现细节 |
| 代码审查 | 从实现质量、需求一致性上做技术收口 |
| 测试验证 | 从功能正确性、稳定性上做结果收口 |

**模型分配建议**：
- PM：简单模型（只做路由）
- 专业角色（需求分析、方案设计、代码审查等）：强模型

### 2.5 第五步：补流程定义文件和角色契约

**流程定义文件**：
- 有哪些阶段
- 每阶段由谁负责
- 正常怎么往前走
- 出问题允许退回到哪

**角色契约**（每个 Agent 的接口说明）：
- 必须读哪些上游产物
- 必须写出哪些文档
- 什么情况下可提出阻塞
- 什么情况下必须交回 PM

### 2.6 第六步：补 dev-map 和任务看板（持续迭代时）

**dev-map（开发导航地图）**：
- 功能落在哪些文件
- 服务怎么接入
- 配置在哪定义
- 改一个模块会牵动哪些链路
- 已有的标准写法

**任务看板**：
- 当前有哪些任务
- 每个任务进行到哪个阶段
- 对应文档目录在哪
- 已完成任务的交付结论

### 2.7 第七步：考虑 MCP（闭环延伸到外部系统时）

**适用场景**：
- CI 构建
- 签名服务
- 制品上传
- 发布系统
- 状态回写

## 三、落地过程中的常见问题与解法

### 3.1 下游自己修改上游文档
**问题**：方案设计私自修改需求，责任边界模糊  
**解法**：下游不能直接改上游文档，只能提出阻塞项，由 PM 正式打回

### 3.2 PM 太容易越界
**问题**：PM 从流程管理者变成到处给意见的人  
**解法**：PM 只负责流程管理，不负责专业判断，不给建议

### 3.3 专业性不够深
**问题**：代码审查和测试只做常规检查  
**解法**：
- 代码审查必须对照需求和方案确认完整性
- 测试要看功能、边界、回归、稳定性

### 3.4 Rule 约束不够用
**问题**：AI 解释性执行、找借口绕过  
**解法**：把能判定的约束落地成脚本

### 3.5 AI 偷懒
**问题**：AI 说"这个错误本来就存在"  
**解法**：开发前后做基线对比，新增错误必须修

### 3.6 流程本身难维护
**问题**：规则都在长文里，阶段多了后双写不一致  
**解法**：
- 把流程结构抽成流程定义文件
- 给每个 Agent 补角色契约
- 补流程校验脚本（检查文件是否存在、契约是否齐全）

## 四、总验证脚本检查清单

### A类：静态规范
- XAML 中文/Emoji 检查
- C# 8.0+ 语法检查
- MessageBox.Show 使用检查
- 硬编码 UI 文本检查
- 日志前缀检查
- 文件长度检查
- 本地化合规检查

### B类：基础交付
- 编译必须成功
- 测试必须全部通过
- 测试数量不能异常减少

### C类：工程一致性
- 规则文件多格式同步
- .cs 文件是否进了 .csproj

## 五、四块拼图模型

| 拼图 | 作用 | 对应内容 |
|------|------|---------|
| 约束与流程 | AI 按什么顺序、边界做事 | Rule、Skill、多 Agent、Workflow |
| 反馈 | 对结果给出可执行判定 | 总验证脚本、事后验证 |
| 知识库 | 项目级进门索引 | dev-map、任务看板 |
| 进化 | 人与 AI 分工，持续改版 | 规范沉淀、责任划分 |

**四块关系**：
- 只有流程没有反馈 → 完成幻觉
- 只有反馈没有流程 → 上游一团即兴
- 没有知识库 → 重复造轮子
- 没有进化 → 三块冻在某版无法迭代

**核心图景**：
> 用流程和约束把动作摆进轨道，用反馈让轨道尽头有人判卷，用知识库让 AI 别在城里瞎撞；进化则是人与 AI 合力，把实践与门禁里长出来的结论持续灌回流程和约束、反馈与知识库——整套东西才真正跟项目活下去。

## 六、人与 AI 的分工

| 角色 | 职责 |
|------|------|
| **人** | 定义目标和边界、设计流程、制定规则、设计门禁、审视漏洞、最终判断 |
| **AI** | 在系统里高强度执行 |

> AI 不是助手，更像一支执行力极强但必须被制度化管理的团队。

---

## 七、关于 Memory 的原则

| 类型 | 该落在哪 |
|------|---------|
| 定了的结论、边界 | SPEC、任务看板、设计说明 |
| 成功/失败样例 | 测试用例、门禁脚本、评审清单 |
| 反复踩的坑、红线 | 优先 Scripts，其次 Rule |

**核心原则**：团队要对齐的东西，应落在仓库里能看见、能审计、能交接的资产上，而不是停在会话记忆里。

---

## 八、快速起步路径

```
1. 磨出设计规格文档 → 目标、边界、验收标准
2. 补关键 Rule → 最容易出错的底线
3. 做成 Skill → 编译、测试、验证
4. 拆多 Agent → 单 Agent 失稳时
5. 补流程定义和契约 → 阶段多、回退多时
6. 补 dev-map 和看板 → 持续迭代时
7. 考虑 MCP → 闭环延伸到外部系统时
```

> 不要贪大，不要一步到位，先从最痛的那个问题开始。

## 九、Harness 全貌

```
设计规格文档（SPEC）→ 目标
       ↓
     Rule → 底线
       ↓
     Skill → 标准动作
       ↓
   Sub Agent → 角色分工
       ↓
   Workflow → 接力规则
       ↓
   Scripts → 验收闭环
       ↓
  dev-map / 任务看板 → 项目地图与态势
       ↓
     MCP → 外部系统连接
```

> Harness Engineering 就是给 AI 搭一整套"工程作战系统"：SPEC是作战目标，Rule是纪律，Skill是标准动作，Sub Agent是兵种分工，Workflow是指挥链，Scripts是验收闭环，dev-map和任务看板是地图与态势。

## 十、落地核心原则

1. **不要一次搭齐**：先用最小可用形态跑起来，在真实问题里不断补结构、补边界、补门禁、补反馈闭环

2. **哪痛补哪**：先看当前最大的痛点在哪里，先补最关键的一层，让系统先跑起来

3. **规则继续脚本化**：只要是能判定的，就尽量不要停留在 Rule 层，下沉成脚本

4. **知识沉淀进项目**：不应该越来越像"我和我的 AI 的默契"，而应该是"任何人拿到这个工程都能顺着这套系统做对事情"

## 十一、关键模块速览表

| 模块 | 在 Harness 中的作用 |
|------|-------------------|
| 设计规格文档 | 统一版本目标、边界和实现方向 |
| Rule | 写死基础规范与底线 |
| Skill | 把固定动作标准化 |
| 总验证脚本 | 建立事后验证和反馈闭环 |
| 多 Agent 定义 | 固定角色分工 |
| 流程定义文件 | 固定阶段与迁移关系 |
| 角色契约 | 固定每个角色的输入输出边界 |
| 开发导航地图 | 帮助 AI 快速理解项目结构和已有模式 |
| 任务看板 | 给 PM 和需求分析提供全局视角 |
| 多阶段文档链 | 让复杂需求有完整过程留痕 |
